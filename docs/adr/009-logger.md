# ADR-009: logger

- **Status**: Accepted
- **Date**: 2026-04-27

---

## Context

서버 운영 중 동작 상태를 추적하고, 환경별 로그 출력을 제어하기 위해 로그 라이브러리가 필요하다.

선택 가능한 라이브러리와 특징에 대해 아래 표로 정리하였다.

|   | slog | zap | zerolog |
|---|---|---|---|
| 의존성 | 표준 라이브러리 | 외부 | 외부 |
| 성능 | 보통 (~100 ns/op) | 빠름 (~51 ns/op) | 가장 빠름 (~25 ns/op) |
| 생태계 | 표준이라 어댑터 풍부 | 가장 많이 사용 | 활발 |

## Decision

**slog를 선택**

## Rationale

### slog 선택 이유

- 표준 라이브러리이기 때문에 외부 의존성과 공급망 리스크가 없다.
- 인증 서버 기준으로 트래픽 기준 성능이 충분하다.
- 핸들러 교체로 다른 로그 라이브러리를 갈아 끼울 수 있다. 애플리케이션 코드를 `*slog.Logger`에 의존시켜 두면, 추후 성능 이슈 발생 시 코드 수정 없이 zap 같은 다른 라이브러리로 교체 가능하다.

### 기타 라이브러리를 선택하지 않은 이유

- **zap**: 가장 널리 사용되지만, `slog`로도 성능상 충분하기에 외부 의존성을 가져올 필요가 없다.
- **zerolog**: 성능은 가장 우수하나, 위와 동일하게 외부 의존성을 가져올 필요가 없다.

### 한계

- `TRACE`, `FATAL` 레벨이 기본 제공되지 않아 커스텀 정의가 필요하다.
- 샘플링, 테스트용 observer 같은 고급 기능이 빠져있어 필요 시 직접 구현하거나 커뮤니티 패키지에 의존해야 한다.

## Consequences

- 커스텀 에러 레벨 정의(e.g. TRACE, FATAL, ...)
- 성능 상 문제가 발생하는 경우, 로그 핸들러를 교체할 가능성 존재
