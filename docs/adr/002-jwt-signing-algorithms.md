# ADR-002: JWT 서명 알고리즘 HS256 선택

- **Status**: Accepted
- **Date**: 2026-03-10

---

## Context

JWT access token의 서명 알고리즘을 결정해야 했다.
주요 선택지는 두 가지였다.

- **HS256 (HMAC-SHA256)**: 대칭 키. 서명과 검증에 동일한 secret을 사용.
- **RS256 (RSA-SHA256)**: 비대칭 키. private key로 서명, public key로 검증.

---

## Decision

**HS256을 사용한다.**

---

## Rationale

### HS256을 선택한 이유

이 서버는 단일 서비스이며, 토큰을 서명하는 주체와 검증하는 주체가 auth-server 하나다.

RS256의 핵심 장점은 public key만 배포하면 외부 서비스가 auth-server를 거치지 않고 토큰을 직접 검증할 수 있다는 점이다.
그러나 현재 구조에서는 토큰 검증 주체가 여러 서비스로 분산되어 있지 않으므로 이 장점이 필요하지 않다.

**단일 서비스**에서 HS256은 다음 이점을 가진다.

- 서명과 검증이 같은 서버에서 이루어지므로 키 공유 문제가 없다
- 키 관리가 단순하다. secret 하나를 Secret Manager에 보관하면 된다
- RS256 대비 구현과 운영이 단순하다

### RS256을 선택하지 않은 이유

RS256은 키 페어 생성, 순환(rotation), public key 배포 등 관리 부담이 있다.
현재 구조에서 이 복잡성을 감수할 이유가 없다.

---

## Consequences

- JWT secret은 GCP Secret Manager를 통해 주입한다. 코드나 환경변수 파일에 직접 포함하지 않는다.
- 토큰 검증 주체가 여러 서비스로 확장되는 경우 **RS256으로 전환을 검토해야 한다.** HS256에서는 secret을 공유해야 하며, 공유 범위가 넓어질수록 키 유출 리스크가 커진다.
