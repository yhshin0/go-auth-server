# ADR-003: 세션 저장소 Redis 선택

- **Status**: Accepted
- **Date**: 2026-03-10

---

## Context

Refresh token 및 세션 데이터의 저장소를 결정해야 했다.
주요 선택지는 두 가지였다.

- **PostgreSQL**: DB. 영속성 보장.
- **Redis**: 인메모리 저장소. TTL 내장, 고속 조회.

---

## Decision

**세션 데이터는 Redis에 저장한다.**

---

## Rationale

### Redis를 선택한 이유

**조회 성능**

세션 데이터는 인증 요청마다 조회된다. refresh 요청 시 `rt:{hash}` → `sess:{sid}` 두 번의 조회가 발생하며, 이 경로가 느리면 인증 전체가 느려진다.
인메모리 저장소인 Redis는 DB 대비 조회 지연이 낮고 예측 가능하다.

**TTL 자동 만료**

Redis는 키 단위 TTL을 내장하고 있다. 세션 만료를 별도로 구현할 필요가 없다.
DB를 사용했다면 만료된 세션을 주기적으로 정리하는 배치 작업이 필요했을 것이다.

**영속성이 필요 없는 데이터**

세션의 원본 데이터(유저 정보)는 PostgreSQL에 있다. Redis의 세션이 소실되더라도 데이터 손실은 없으며, 유저는 재로그인으로 새 세션을 발급받을 수 있다.
세션은 본질적으로 임시 데이터이므로 영속성을 강제할 이유가 없다.

### PostgreSQL을 선택하지 않은 이유

세션 조회는 빈도가 높고 응답 지연에 민감하다. DB에 세션을 저장하면 인증 트래픽이 애플리케이션 DB에 그대로 부하로 이어진다.
또한 만료 세션 정리를 위한 별도 배치 작업이 필요해 운영 복잡도가 높아진다.

---

## Consequences

- Redis restart 또는 장애 시 전체 세션이 소실되어 모든 유저가 강제 로그아웃된다. 데이터 손실은 없으며, 세션 미존재 시 401을 반환해 클라이언트가 재로그인을 하도록 유도한다. 이것이 휘발성에 대한 대책이며, 현재는 이 수준으로 충분하다고 판단한다.
- Redis persistence(AOF/RDB) 설정은 현재 적용하지 않는다. 트래픽 증가 또는 세션 소실이 민감한 문제가 되는 시점에 재검토한다.
- 세션 데이터 구조(`sess:{sid}`, `rt:{hash}`, `user_sessions:{userId}`)는 Redis에만 존재한다. 세션 이력이 필요한 경우 `auth_events` 테이블을 참고한다.
