# ADR-004: 유저당 세션 제한

- **Status**: Accepted
- **Date**: 2026-03-10

---

## Context

다중 기기 로그인을 허용할 경우 유저당 세션 수에 상한을 둘지 결정해야 했다.
선택지는 세 가지였다.

- **단일 세션**: 새 로그인 시 기존 세션 무효화. 보안은 단순하지만 다중 기기 사용이 불가.
- **무제한 세션**: 기기 수 제한 없음. UX는 좋지만 세션 관리 부담이 있음.
- **상한 설정**: 다중 기기를 허용하되 세션 수를 제한.

---

## Decision

**유저당 최대 5개의 세션을 허용한다. 초과 시 가장 오래된 세션을 자동 제거한다(LRU).**

---

## Rationale

### 무제한을 선택하지 않은 이유

세션은 Redis에 저장되며, 세션 수가 무제한이면 Redis 메모리가 유저 수에 비례해 계속 증가한다.
또한 탈취된 토큰이 오랫동안 방치되어도 감지가 어렵다. 세션 수에 상한을 두면 자연스럽게 오래된 세션이 정리되어 메모리와 보안 모두 관리가 쉬워진다.

### 단일 세션을 선택하지 않은 이유

모바일, PC, 태블릿 등 여러 기기에서 동시에 사용하는 것이 일반적인 사용 패턴이다. 단일 세션은 UX를 불필요하게 제한한다.

### 5개로 설정한 이유

모바일, 태블릿, 개인 PC, 직장 PC, 여분 기기 정도를 커버하는 현실적인 기기 수다.
다중 기기 로그인을 허용하는 서비스들의 일반적인 범위(3~10개) 안에 있으며, 현재 서비스 규모에서 합리적인 기본값으로 판단했다.

### LRU 방식(가장 오래된 세션 제거)을 선택한 이유

5개 초과 시 가장 최근에 사용한 세션을 유지하고 가장 오래된 세션을 제거하는 것이 사용자 입장에서 가장 자연스럽다. 현재 사용 중인 기기가 갑자기 로그아웃되는 상황을 최소화할 수 있다.

---

## Consequences

- 상한값 5는 설정값으로 관리하는 것이 이상적이나, 현재는 코드 상수로 고정한다. 서비스 성격에 따라 조정이 필요할 수 있다.
- LRU 제거 시 `sess`, `rt`, `user_sessions` 세 곳을 원자적으로 정리해야 한다. `sess:{sid}`가 이미 만료된 경우 `ZREM`으로 목록만 제거한다.
- 세션 수 초과로 제거된 기기는 별도 알림 없이 강제 로그아웃된다. 필요 시 `auth_events`에 별도 이벤트 타입 추가를 검토할 수 있다.
